# Development Playbook (the EXECUTION layer / SKILL)

> This is the **entry point and the thin checklist**. The full model, the reason each rule is the way it is, every term, and the pass/done criteria themselves all live in the sibling **`METHODOLOGY.md`** (the single source of truth). This file keeps only the few inline reminders an executor needs mid-run (the brainstorm-split trigger, the approval-point trigger, the L2 review gate, the L1 pass bar); the authoritative full criteria live in `METHODOLOGY.md`, and each step points back to the section that holds them.

## When to use / not use

- **Scope = MEDIUM and above.** A MEDIUM task is a single clear requirement that must land as running code, up to roughly one service / module, carried as the contract `{requirement, done-check}`. The shape of the requirement (build a framework / add a feature / fix a bug / refactor / improve a metric / optimize performance / ...) does not matter. What decides "use it or not" is **complexity**, not lines changed: use it when the work spans multiple coupled modules or a pipeline and is easy to get subtly wrong. Full rationale and the LOW / MEDIUM / above-MEDIUM tiers are in `METHODOLOGY.md` §0.5.
- **Do not use it** for a LOW-complexity task — a single simple file or function (for example a basic random-number generator), a change you can verify just by reading it, or a pure question / explanation / technology-choice consultation (the output is an answer, not code). The checkable test for a one-line change: if you can confirm it correct just by reading it, skip the method; if confirming it needs reasoning through other code paths or writing a test, use the method.
- **Above-medium IS in scope — it just starts with the brainstorm-split.** A COMPLEX task (larger than one service, for example a whole multi-service project) is not refused and is not "outside the method." Its **first phase is a brainstorm-split done together with the human**: decompose the complex task into MEDIUM-sized pieces (each one fitting a single run / a single context), then run the method on each piece in dependency order. The split is a **human+agent brainstorm**, not the agent alone — see "Phase 0" below and `METHODOLOGY.md` §0.5 (including why the split must stay human-in-the-loop, and the lead/orchestrator vs Executor roles).

## Phase 0 (complex tasks only): the brainstorm-split into medium pieces

- **This is the method's first phase for an above-medium task, not a step outside it.** When the task is larger than one service, the FIRST thing is a brainstorm between the human and the agent (the lead / orchestrator role) to decompose it into MEDIUM-sized pieces — each fitting one run / one context. This is exactly how a too-large task is brought down to context-fitting medium pieces.
- **Keep it human-in-the-loop.** The split is a human+agent brainstorm, not autonomous AI, for the reasons in `METHODOLOGY.md` §0.5: AI weakens above the direction/approach level, so charting and decomposing a whole project is where it is least reliable; accountability for a large effort must sit with the human; and a whole project does not fit in a single context, which is itself what forces the work down to the medium granularity.
- **Then run the method per piece.** Take the resulting medium pieces in dependency order and, for each one, run the per-medium-task flow below (Step 0 + the outer loop). A MEDIUM task arriving on its own skips Phase 0 and goes straight to the flow. So the overall shape for a complex task is: **brainstorm-split -> [for each medium piece, in dependency order: Step 0 + the outer loop].**

## Standing preconditions (continuously true for the whole medium task, not a one-time t=0 step)

- **A mechanically self-judgeable done-check must exist, and every iteration must end by self-judging against it.** It must already exist before the first line of L1 code is written; what it looks like concretely is set by the requirement shape — look it up in the dispatch table in `METHODOLOGY.md` §5.
- Startup preparation writes only the **first version** of the done-check; all adversarial review happens inside the iterations, bound together with code + tests / measurement. See `METHODOLOGY.md` §8.

## The outer loop (L3-driven, one iteration advances the requirement once)

The model and the per-step "pass this layer" criteria are in `METHODOLOGY.md` §2 and §3; this lists only the order. (For a complex task, this loop runs once per medium piece, after the Phase 0 brainstorm-split above.)

0. **Step 0 (once, before the loop)** — confirm a mechanically self-judgeable done-check exists. If the requirement handed down does not come with one, settle it with the human first, then enter the loop. Without it the outer loop cannot terminate on its own. See `METHODOLOGY.md` §8.
1. **L3 decides the next step** — what to advance toward this requirement next. The reference is the external world: desk research by default; when desk comparison cannot settle it or the stakes are high, take an actual measurement (build a thin spike and measure). Watch the approval point on two counts here (both per `METHODOLOGY.md` §4): a **cheap** spike runs autonomously, but an **expensive** spike is a high-cost action and goes through the approval point first (cost axis) before you spend the compute; and separately, the direction conclusion the measurement produces, if it changes the approach, goes through the approval point in step 2 (top-level-design axis). Do not treat a measurement as automatically exempt from the approval point. Criteria are in `METHODOLOGY.md` §2 "L3 research".
2. **Human approval point** — where the human steps in; it is not one fixed spot. It can fire at two moments (both per `METHODOLOGY.md` §4): **inside L3, before a high-cost action** (the cost-axis gate from step 1, before the spend), and **at the L3 -> L2 handoff, for a change that touches the top-level design**. It only ever gates the downward direction; reporting results upward never passes through it. **Reminder — the trigger has two axes; hitting either passes through: (1) the change touches the top-level design, or (2) the action spends a large cost (money / time / compute). When unsure, treat it as a big change and go through.** What passes, what is autonomous, and how delegation is handled are in `METHODOLOGY.md` §4.
3. **L2 mid-level (autonomous)** — turn this step into a mid-level SDD (pseudocode; the 9-section SDD format is in `METHODOLOGY.md` §2 "The SDD format"); for a larger piece, first slice it along its dependencies (S1 / S2 / ...). The pass criterion is in `METHODOLOGY.md` §2 "L2 mid-level". **Reminder — the review gate is one pass with several (at least two) concurrent, independent, context-clean reviewers, each using a different lens; if they surface no substantive issue, the layer passes.**
4. **L1 implementation (autonomous)** — faithfully write the code from the SDD plus your own tests. **Reminder — the pass bar is: the tests written from the SDD all pass and it actually runs without errors.** Full criterion in `METHODOLOGY.md` §2 "L1 implementation".
   - **If you sliced, there is an inner loop: for each slice S_i, run "step 3 write this slice's SDD -> step 4 write this slice's code + self-judge this slice" once per slice, finishing one slice before starting the next — do not write all the slices' SDDs / code up front.** A small task reduces to a single slice and has no inner loop.
5. **Self-judge "done or not" against the done-check (+ evidence)** -> feed the result back up to L3 (no approval point on the way up). This self-judgment is the same action for every shape; only the form of the evidence changes by shape — see `METHODOLOGY.md` §3.
6. **Go back up one layer** — when a layer's adversarial review surfaces a problem that actually lives in the layer above it (code is wrong -> back to L2; the approach fails on measurement -> back to L3), go back up and redo that layer. The mechanism is in `METHODOLOGY.md` §1.

## Termination (the same for all shapes)

- **The termination criterion is uniform across shapes**: **the done-check is met**, or **L3 judges that within the approved direction there is no remaining step that can advance this requirement.** The done-check itself should have no hole — it too should be passed through adversarial review once. See `METHODOLOGY.md` §3 (termination) and §5 (the neutral reminder).
- **"When there is no next step" is detailed by shape**: for one-shot shapes (build a framework / fix a bug / refactor) termination is almost always driven by the done-check being met, and the second part rarely fires; only for shapes that converge gradually and in principle always have a next thing to try (the typical case being improve a metric) does the second part become a cost-versus-benefit judgment (is further investment still worth the expected benefit) — see `METHODOLOGY.md` §7. The skeleton is uniform; what is detailed per shape is how "no remaining step" is judged.
- For each shape, read "what counts as done" off the corresponding row of the dispatch table in `METHODOLOGY.md` §5.

## The three invariants (expanded in `METHODOLOGY.md`)

1. **The reference chain**: external world -> top-level design -> mid-level design -> low-level code, each layer checked against the one above it as its reference, with adversarial review used to find what does not match. See §1.
2. **The human is only at L3**: direction belongs to the human (the approval point guards top-level-design changes at the L3 -> L2 handoff, and high-cost actions inside L3); for a complex task, the up-front brainstorm-split sits at this same altitude — see Phase 0. L2 / L1 are fully autonomous. See §4.
3. **The done-check fixes termination**: do not start without a mechanically self-judgeable done-check; self-termination rests entirely on it. See §3 and §8.

## Safety conventions (independent of requirement shape)

- **Changing a production service** (touching a running daemon / live data) must go through a branch + a test gate + human review before merging to the main branch; **never run a migration on production data** — a migration is tested only on a temporary copy. See `METHODOLOGY.md` §8 (landing conventions).

## Metric-improvement appendix (a pointer; applies only to the "improve a numeric metric" shape)

This shape (the typical case being improve a metric; performance / latency optimization belongs here too) is not a separate method — it is the same general outer loop above, applied to a number. It adds no new skeleton: each iteration L3 finds the next candidate approach worth trying, you triage each by cost (cheap ones autonomous, expensive ones through the approval point of §4), measure each and keep the wins, and stop when the candidates are exhausted or further investment is no longer worth it. **The full loop, and the one plain-words caveat (make sure the metric truthfully reflects the real goal before you optimize it, so the number rising genuinely means the thing got better), are in `METHODOLOGY.md` §7.** Do not apply any of this to other shapes.
