# DevStandard v0.6.1 — Adversarial Review Synthesis

> Research artifact behind v0.7.0 (issues #15–#17). Produced 2026-07-16 by a 20-agent workflow: 6 finder angles → triage (22 raw findings → 11 merged) → adversarial refutation (6 confirmed, 5 refuted by existing doc text) → synthesis. The repo-CLAUDE.md proposal (with the teardown write-back) is PARKED pending the human's decision — deliberately not implemented in v0.7.0.

Six angles ran (method-friction, walkthrough, field-compare, failure-modes, ecosystem, pointer-audit); 22 raw findings were merged to 11 candidates; each survived or fell under adversarial verification against the actual files. This report uses the adversary's `revised` form wherever one exists.

## TL;DR

- **Q1 — What can be optimized:** Five fixes survived, all lean. The two sharpest are mechanical dead-ends the docs walk you into: (a) once howto/cicd.md's recommended branch protection is on, **three documented moments require direct commits to main that GitHub will reject** (the "do it right here" lane, the spec status flip, and all setup commits); (b) **cicd.md:33 recommends GitHub's merge queue, which structurally violates core.md/ADR 0011's reviewed-diff-is-merged-diff rule** — a setup-time instruction that installs a standing check-1 bypass. Plus two token-cheap consistency fixes on the injected page (delete the contradictory merge-authority parenthetical at core.md:78; widen two brief-delivery clauses to cover workflow agents).
- **Q2 — What we never thought of:** One genuine blind spot: **DevStandard is structurally amnesiac.** Clean context is the design (fresh workers, fresh reviewers), so written memory is the only memory — yet setup never creates a repo CLAUDE.md (build/test/run commands, gotchas — the field's single most-converged practice), and no merge-time write-back codifies lessons, so every repeatable mistake is repeated and re-caught at review cost forever. Fix costs zero core.md tokens.
- **Q3 — Coupling candidates beyond superpowers:** **None.** The ecosystem survey (six installed official plugins + top-installed 2026 plugins) found zero new pointers that earn their place: tool plugins (context7, firecrawl, frontend-design) are self-triggering mechanics; process plugins (official code-review, feature-dev) are competitors on owned territory. The one surviving action is defensive: generalize core.md L35's sovereignty clause so this-page-wins applies to skills **from any plugin**, not just superpowers (~10 tokens).
- **Q4 — Pointer coverage:** **Covered.** All four pointable superpowers skills sit at their correct steps in the right docs; no never-point skill leaked in. One delivery seam survived verification (workflow-agent workers are outside the worker-brief paste instruction); the other (mini-setup howto routing) was refuted.
- **Batch plan:** Three issues with disjoint file sets, parallel-dispatchable, followed by one release call.

## Rejected on review (for transparency)

| Rejected proposal | One-line reason |
|---|---|
| Design-spec lifecycle doesn't compose with issue dispatch (author / challenge timing / status flipper) | The dispatch taxonomy already routes it: subagents only get fully-specified work (spec exists and is linked, core.md:62), separate sessions get core.md itself and are design-spec.md:30's intended audience; the worker's stop-and-escalate list fences the mis-dispatch case, and naming a flipper for a no-handoff status change is ceremony. |
| Adoption procedure for existing branches/worktrees (post-merge rebases; dead-worker re-dispatch) | Every claimed failure is blocked by a cited existing line: merged-result CI (core.md:74) absorbs clean cases with no rebase at all, conflicted rebases already have a named owner (worker-brief:31 fresh issue), the explicit-cancel gate (core.md:80) is the two-writers guard, and Birth's passing-start check prevents building on half-written state — an "Adoption" block would be pure restatement. |
| Bug issues need a done-check only the worker can produce | core.md:17 already names "the bug no longer reproduces" as a valid done-check the main session can state from a pinned symptom, and the repro-test-fails-then-passes form is verbatim superpowers TDD + systematic-debugging craft — restating it in core.md violates the sovereignty split and eats the whole headroom. |
| No rule for dependent tasks (task B needs unmerged task A) | Triple-fenced already: the base is explicit in three places (never another task's branch), the worker escalation path covers reporting the dependency, rebase-re-runs-check-1 independently guards the invariant, and interface-first-as-its-own-PR is the method's already-taught skeleton idiom. |
| False-Critical deadlock: no dismissal path or round cap on check 1 | The gate blocks on the latest fresh review, not standing per-finding debt — independent resampling dissolves hallucinated Criticals for free, two independent reviewers agreeing means the block is working, the interactive human IS the round cap, and an evidence-dismissal channel invocable by the anchored main session reintroduces exactly the self-grading ADR 0011 rejected. |
| Issue mutated/cancelled under an in-flight worker: no freeze rule | The amendment channel already exists (changes count only once written to the issue/PR, core.md:68), cancellation is explicit-only (core.md:80), and divergence fails safe — check 1 blocks the mismatched PR; a permanent rule for a rare, human-caused, fail-safe scenario doesn't earn its cost. |
| CI gate is tests-only; field baseline adds lint/typecheck (dropped at triage) | Lowest value-per-slot: toolchain-dependent, no dead-end without it, slow self-evident failure mode — rides along as a one-line template comment in the cicd.md issue, no more. |

## Pointer-audit verdict

Verbatim from the pointer-audit angle:

> "I walked every step of DevStandard's flow as written — full setup (PRD, architecture, ADR log, skeleton, CI/CD, task split), one task (done-check, spec gate, executor choice, challenge, build, evidence, rebase, PR), integration (check 1, CI, merge, teardown), plus each howto and aid as a reading moment — and checked at each step whether a pointable superpowers 5.1.0 skill serves it from the doc the reader actually has open. All four pointable skills sit at their correct steps: brainstorming (core.md L35 canonical + step-local redirects in howto/prd.md L5 and howto/architecture.md L5), systematic-debugging and TDD (core.md L35 + the intentional worker-brief L20 mirror), writing-plans (howto/design-spec.md L23 only, fully fenced, with core.md L19 as the discovery path). No pointer is at a wrong step, no never-point skill has leaked in, and no unserved step wants a pointable skill. The two residual seams are delivery seams, not missing pointers: workflow-agent workers fall outside the worker-brief paste instruction (so the only craft pointers a non-session worker gets may never reach a level-3 implementer), and the mini-setup line never routes its reader to the howtos that carry brainstorming's save-path redirect. VERDICT: covered — no missing, misplaced, or wrong-doc skill pointers; two low-cost delivery seams found (workflow-agent worker-brief delivery; mini-setup howto discovery), listed below."

**Surviving gap after adversarial verification:** only the workflow-agent delivery seam (surviving proposal #6). The mini-setup seam was refuted — core.md is injected whole, L11 names the howtos with a moment-keyed reading rule on the same page, and the brainstorming save-path hazard is already held by the redirect lines inside howto/prd.md.
