# DevStandard(core.md 中文对照)

> 本文件供人阅读。插件注入 session 的是英文原版 [`core.md`](core.md),两者内容一致,以英文版为准。

**为什么要有这套:**开发大项目时,配合 coding agent 最快的方式,是让多个目标并行推进——多个功能、多个 bug 同时往前走,而不是一件接一件。下面这一切——分支、worktree、PR、门——只为一个目的存在:让这种并发安全地成立。单独一件小改动用不着这些,直接做就行。

下面讲怎么运作:完整生命周期何时启用、一个任务怎么执行、并行工作怎么协同。模板和辅助工具都在本插件里——需要时才读,绝不预读。

## 触发

- 人要求**起一个新项目**(一个新仓库,或 monorepo 里一个新的顶层 package/app/service)→ 启动完整生命周期:
  PRD → 架构文档 + ADR 日志 → 薄骨架(接口/边界落成代码,钉死并行工作的接入位)→ CI + 发布流水线 → 拆任务、派出去。
  到哪个产物再读 `howto/prd.md`、`howto/architecture.md`、`howto/adr.md`、`howto/cicd.md`——别提前读。人不吭声默认就是全套;绝不擅自降级。
- 人声明它小(随手 / 实验 / 草稿 / 配置)→ 轻启动:只要 CI,或什么都不要。人日后随时可以说"升级到全套"。
- 已有仓库里的改动 → 通常就是一个任务(规则见下)。但人声明为大工程的仓库内倡议——或触及顶层设计、或花费巨大的——走迷你生命周期:带自己完成标准的 PRD 增量、架构文档更新 + ADR、拆任务,然后走正常流程。

## 执行一个任务

**写任何代码之前:先定下机器可自判的完成标准**——什么能机械地证明这个任务做完了(测试通过 / bug 不再复现 / 指标动了)。需求含糊 → 先和人定清楚。

**挑能托住这个活的最便宜档位:**
1. session 里直接干——多数工作的默认。
2. 1–3 个干净上下文的子 agent——有独立的块、或独立评审有帮助时;无循环、无宽扇出(子 agent 可以继续分包——对一个块的深度支援仍算本档)。
3. 一发小 workflow——只为真实的扇出(评审团)或真实的循环(修到全绿)。
4. 多发链式 workflow——工作跨越了决策点。

**每个档位都守的纪律:**
- 非平凡的设计,实现之前至少被一个干净上下文的评审驳过。
- 一个 worktree 只有一个写者——绝不让两个 agent 同时改同一批文件。(不同 worktree 并行推进正是我们要的;禁的只是对**同一份**代码的并发编辑。)一个任务内部,并行花在评审/验证上,不是加第二个写者。
- "做完了"必须带证据:命令、退出码、输出。评审者失联算失败,不算通过。
- 只在改动触及顶层设计、行动花费巨大、或行动具有破坏性/难以回退(删数据、force-push、任何离开仓库的动作:发布、外发)时问人;其余自治。拿不准就当大事,问。

**workflow(档位 3–4):**一发 run 是一段盲飞的完整阶段——运行中无人能干预。发射前把花费封顶:评审团人数写死、每个循环带轮数上限、预算守卫。run 在决策点/检查点处拆,绝不因容量拆;run 之间靠磁盘上的 commit 和文档接力。模型按角色路由:判断/综合 → 可用的最强档;评审团 → 降一档;机械活 → 降两档。绝不从顶配 session 发射宽扇出——agent 会继承 session 的模型。

## 协作标准

agent 团队像人类 GitHub 团队一样运转:issue 派活、PR 交活、main 受守护。开工前读仓库的 `docs/architecture.md`(共同底图),扫一眼 `docs/adr/`。

**main session 是驾驶舱**(你 + 人):核心讨论、项目定义、需求 drill、派活、集成都在这儿发生。它把活派出去、也收回来——外层唯一的枢纽。

**人不碰 git——全交给 agent。**每一次 commit、push、开分支、merge、打 tag,都由 agent 来执行。人只管方向(要什么结果、为什么)和放行(动架构、发版的 go/no-go)——管决定,不管敲键。

**派活 = 一个 GitHub issue——驾驶舱在派出去之前,只需钉死两件事:你要一个什么结果,以及为什么。**定清目的和结果,"怎么做"整个留给 worker。用什么架构、怎么实现,是 worker 的判断——过度规定方法既浪费它的判断力,也不是驾驶舱该操心的。issue 就是那份可追溯、可查的规格:想要的结果 + 机器可自判的完成标准。值得单开一条分支的活走 issue;琐碎改动跳过、会话里直接做。open issues + open PRs 就是驾驶舱的全部待办——只凭 GitHub 就能重建;没有要紧的东西只活在某个 session 脑子里。

**执行者由阶梯挑最便宜那档。**琐碎 → 驾驶舱会话里直接干(不开分支)。任何值得开一条分支的活 = 一条分支 = 一个 worktree,由:嚼烂且有界 → 驾驶舱派的子 agent 或 workflow;嚼不烂需边做边引导 / 长时间跨天并行 / 别人的活 → 一个单独的 live session。

**如果你是 worker(子 agent、workflow 里的 agent、或单独 session),你的角色与边界:**只有当你是人那个唯一持久的 main session 时,你才是驾驶舱;任何为执行某个 issue 而生的 session 或 agent 都是 worker——默认自己是 worker。
- 你只拥有一条分支、一个 worktree。同一时刻只有一个写者:你派的任何帮手只做验证/评审——只读、不给它自己的 worktree。你绝不执行合并——那是驾驶舱的动作。
- 该做:只在自己的分支/worktree 里干;实现那个已被反驳通过的设计;交付前 `git fetch` 并 rebase 到当前 main、自己解掉冲突;跑完成标准、留下证据(命令、退出码、输出);推分支、开一个挂在 issue 上的 PR。
- 绝不:merge 到 main;推发布 tag;碰任务外的文件;改别人的分支;为了让完成标准通过而改弱、跳过或删掉它;无证据就宣称做完。
- 上报驾驶舱(别自己拍):任务原来触及核心架构;需要破坏性/难回退的动作;完成标准错了或够不着、或设计需实质变更;卡在方向决策上。
- 完成:一个挂在 issue 上、已 rebase 到当前 main、描述里带证据的 PR。集成(评审 + 合并)是驾驶舱的事,不是你的。
- (子 agent 收不到注入——驾驶舱派活时塞给它 `aids/worker-brief.md`;单独 session 从本页就能看到。)

**集成是驾驶舱的动作,它当决策者。**merge 前两道门,按序:
1. **一个全新的干净上下文 reviewer**——喂给它 diff + issue + 实现者的报告(当作未验证的声明),别的都不给(不给会话历史)。它必须没写过这份代码、也不能是驾驶舱自己攒满的上下文(每次 merge 现拉一个干净子 agent)。它审测试看不见的:diff 对不对得上 issue、测试是不是真的且没被改弱、设计稳不稳。Critical/Important 发现阻塞 → 修 → 复审。(`aids/code-review-prompt.md`)
2. **对当前 main 的合并结果 CI 全绿**——确定性的、无人自评的最后一句话。

被评审的 diff 必须就是被合并的 diff:门 1 通过之后任何 rebase(解冲突,或 base 动了)都要对新 diff 重跑门 1——merge queue 只用于无冲突的快进,绝不用来自动 rebase 绕过评审。

是分层不是替代。然后驾驶舱(判断题交给人:够不够格?动没动架构?)合并、关掉 issue。**merge 归 main——worker 绝不合并,由驾驶舱执行。发版是人的决定,但 tag 和 push 由 agent 来敲。**

**worktree 与任务同生共死。**PR 合并后(或人放弃后):确认没有未提交、未推送的内容,然后从仓库根部依次:移除 worktree、删除分支、`git worktree prune`。绝不清理不是自己创建的 worktree;合并的人顺手清扫孤儿。(`aids/worktree-lifecycle.md`)

**要动核心架构?**绝不悄悄动:摆到台面 → 公开 merge → 人批准 → 更新 `docs/architecture.md` + 写一篇 ADR。你认为底图错了,也走同一条路挑战——绝不闷头照着错的做。

**安全底线:**改动线上服务必须走分支 + 门 + 人工评审;迁移只有在副本上演练过、走门控路径、且手里有验证过的回滚之后,才碰生产。

可选辅助:`aids/worker-brief.md`、`aids/code-review-prompt.md`、`aids/worktree-lifecycle.md`、`aids/testing-anti-patterns.md`、`aids/debugging-techniques.md`。
