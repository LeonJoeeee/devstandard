# DevStandard(core.md 中文对照)

> 本文件供人阅读。插件注入 session 的是英文原版 [`core.md`](core.md),两者内容一致,以英文版为准。

**为什么要有这套:**开发大项目时,配合 coding agent 最快的方式,是让几个目标并行推进——几个功能、几个 bug 同时往前走,而不是一件接一件。下面这一切——分支、worktree、PR、合并前的两道检查——只为一个目的:让这种并行安全地成立。单独一件小改动用不着这些,直接做就行。

下面讲:完整流程什么时候启用、一个任务怎么做、并行的活怎么协同。模板和辅助文件都在本插件里——需要时才读,绝不提前读。

## 什么时候启用完整流程

- 人要求**起一个新项目**(一个新仓库,或 monorepo 里一个新的顶层 package/app/service)→ 走完整流程:
  PRD → 架构文档 + 决策日志(ADR)→ 最小的初始骨架(接口和边界先落成真代码,把并行任务的接入点钉死)→ CI + 发布流水线 → 拆成任务、派出去。
  到哪个产物再读对应的 `howto/prd.md`、`howto/architecture.md`、`howto/adr.md`、`howto/cicd.md`——别提前读。人没说大小,就默认走完整流程;绝不擅自缩水。
- 人说它小(随手 / 实验 / 草稿 / 配置)→ 轻启动:只要 CI,或什么都不要。人日后随时可以说"升级到完整流程"。
- 已有仓库里的改动 → 通常就是一个任务(规则见下)。但人明说是大工程的、或触及顶层设计的、或跑起来很费的仓库内改动,走一个迷你流程:带自己完成标准的小 PRD 增补、架构文档更新 + 一篇 ADR、拆任务,然后走正常流程。

## 做一个任务

**写任何代码之前:先定一个完成标准**——一个机器能自己判定通过/不通过的检查,证明这个任务做完了(测试通过 / bug 不再复现 / 指标动了)。需求含糊 → 先和人定清楚。

**挑能干成这活的最便宜那一档:**
1. 就在当前会话里直接做——多数活的默认。
2. 1–3 个全新、无历史的子 agent——有独立的一块、或独立评审有帮助时;不带循环、不一次开很多个(子 agent 可以再往下分——对一块的深度支援仍算这一档)。
3. 一发小 workflow——只在确实要一次并行开很多个 agent(比如评审团),或确实要个循环(反复修到测试全过)时才用。
4. 多发链式 workflow——活跨越了决策点。

**每一档都守的规矩:**
- 非平凡的设计,动手之前必须扛过一次挑刺:至少一个全新、无历史的评审者主动来挑毛病,没挑出阻塞问题,你才实现——只实现扛过挑刺的那一版。
- 一个 worktree 只有一个写者——绝不让两个 agent 同时改同一批文件。(不同 worktree 并行推进正是我们要的;禁的只是同时改**同一份**代码。)一个任务内部,并行的力气花在评审/检查上,不是加第二个写者。
- "做完了"必须带证据:命令、退出码、输出。评审者没给出结论(空的、报错、超时)算失败,不算通过。
- 只在下面几种情况问人:改动触及顶层设计、这一步很费(比如要发一个 workflow、或一次开很多并行 agent)、或者动作有破坏性/难撤回(删数据、force-push、任何离开仓库的动作:发布、外发)。其余自己做主。拿不准就当大事,问。

**workflow(第 3–4 档):**一发 run 是一段一口气从头跑到尾、中途没法插手的阶段。发之前先把花费封顶:评审人数写死、每个循环给硬性轮数上限、设花费上限。run 在决策点/检查点处拆,绝不只为多干拆;run 之间靠磁盘上的 commit 和文档接力。模型按角色、按相对强弱路由(从强到弱;具体是哪个模型是你自己的配置):判断/综合 → 手上最强的;评审团 → 降一档;机械活 → 降两档。绝不从"正跑着你最强模型"的会话里一次开很多 agent——它们会继承那个会话的模型。

## 一起协作

这个团队像人类的 GitHub 团队一样运转:issue 派活、PR 交活、main 受保护。开工前先读仓库的 `docs/architecture.md`(共同基准),扫一眼 `docs/adr/`。

**有一个会话是"主会话"**(你 + 人):核心讨论、定义项目、逐条问清需求、派活、合并,都在这儿发生。它是所有活派出去、又收回来的唯一一个集散点。

**人不碰 git——全由 agent 来敲。**每一次 commit、push、开分支、merge、打 tag,都由 agent 执行。人只管方向(要什么结果、为什么)和放行(动架构、发版的 go/no-go)——管决定,不管敲键。

**派活 = 一个 GitHub issue。主会话在派出去之前,只需钉死两件事:你要什么结果,以及为什么。**定清目的和结果,"怎么做"整个留给 worker——用什么架构、代码怎么写,是 worker 的判断,过度规定方法只会浪费它。issue 就是那份留得住、可复查的规格:想要的结果 + 一个机器可自判的完成标准。

**哪种改动要单独开分支:**满足下面任一条 → 开一个 issue 和一条分支(一个 worktree):它需要自己的、可能失败的完成标准;动了多个文件、或一个共享/对外的接口;要无人值守跑、或和别的活并行跑;或者不是一句 `git checkout` 就能安全撤回的。否则就是小改——就在当前会话里直接做,不开 issue、不开分支。

open 的 issue + open 的 PR 就是主会话的全部待办——所以状态只凭 GitHub 就能重建;没有要紧的东西只活在某个会话的记忆里。

**谁来干这活:**挑能干成的最便宜那一档。小 → 主会话当场做(不开分支)。要开分支的活 = 一条分支 = 一个 worktree(仓库的一份独立工作副本,在自己的分支上),交给:说得清、边界又定死的 → 主会话派一个子 agent 或 workflow 去做;没法事先说清(过程中会碰到只有人能拍的决定)、或要跨天并行、或是别人的活 → 一个单独的 live session。

**如果你是 worker(一个子 agent、workflow 里的 agent、或一个单独会话),你的角色:**只有当你是人那个唯一、持续的主会话时,你才是主会话;任何为执行某个 issue 而开的 agent 或会话都是 worker——拿不准,你就是 worker。
- 你只拥有一条分支、一个 worktree。同一时刻只有一个写者:你派的任何帮手只做评审/检查——只读,不给它自己的 worktree。你绝不做合并——那是主会话的事。
- 该做:只在自己的分支/worktree 里干;实现那个扛过挑刺的设计;交付前先 `git fetch`、rebase 到当前 main、自己解掉冲突;跑完成标准、留下证据(命令、退出码、输出);推分支、开一个挂在 issue 上的 PR。
- 绝不:merge 到 main;推发布 tag;碰任务外的文件;改别人的分支;为了让完成标准通过而改弱、跳过或删掉它;没证据就说做完了。
- 碰到下面任一种,停下、告诉主会话(别自己拍):任务原来触及核心架构;需要一个破坏性/难撤回的动作;完成标准错了或够不着、或设计要大改;卡在一个方向决策上。怎么告诉它:子 agent 就把这条消息返回给派它的主会话;单独会话就在 issue 上发一条评论(这样它留在 GitHub 里)。人也可以在中途直接跟一个 live 的 worker 会话对话来引导它——但对话里定的任何决定、改的 spec、给的证据,只有写回 issue 或 PR 才算数。
- 你的"做完了":一个挂在 issue 上、已 rebase 到当前 main、描述里带证据的 PR。评审和合并是主会话的事,不是你的。你的 worktree 原地留着——主会话合并时会把它删掉。
- (子 agent 不会自动收到本页——主会话派活时把 `aids/worker-brief.md` 粘给它;单独会话从本页就能看到。)

**合并是主会话的事,它当决策者。**merge 前两道检查,按序:
1. 一个**全新的评审者**(全新开、没有历史)——只给它 diff + issue + worker 的报告(当作未验证的声明),别的都不给(不给会话历史)。它必须没写过这份代码,也不能是主会话自己攒起来的对话(每次合并现开一个全新子 agent)。它审测试看不见的:diff 对不对得上 issue、测试是不是真的且没被改弱、设计稳不稳。标为 Critical/Important 的发现挡住合并 → 修 → 再审。(`aids/code-review-prompt.md`)
2. **对当前 main 的合并结果 CI 全绿**——自动的、不偏袒的最后一句话;它不给自己的活打分。

被评审的 diff 必须就是被合并的 diff:检查 1 通过之后任何 rebase(解冲突,或因为 main 动了)都要对新 diff 重跑检查 1——merge queue 只用于无冲突的快进,绝不用来自动 rebase 绕过评审。

两道检查是叠加的,不是二选一。然后主会话(判断题交给人:够不够格?动没动架构?)合并、关掉 issue。**worker 绝不合并——主会话来做。发版是人的决定,但 tag 和 push 由 agent 来敲。**

**worktree 一做完任务就删掉。**PR 合并后(或人明确取消这个任务——绝不靠"没动静"去猜):先确认没有未提交、未推送的东西,然后从仓库根部依次:移除 worktree、删分支、`git worktree prune`。合并某个 PR 的 agent 负责删掉那个 PR 的 worktree 和分支(哪怕是 worker 建的);对一个既不是你在做、也不是你在合并的任务,别去动它的 worktree。(`aids/worktree-lifecycle.md`)

**要动核心架构?**绝不悄悄动:开一个公开 PR 摆到台面(不是偷偷改)→ 拿到人的批准 → 走两道检查合并 → 更新 `docs/architecture.md` 并写一篇 ADR。批准之前,绝不落到 main。你要是觉得已经定好的架构错了,也走同一条路挑战——绝不闷头写跟它相悖的代码。

**最低安全底线:**改动线上服务,必须走分支 + 两道检查 + 人工评审;迁移只有在副本上演练过、走"评审+CI"这条路、且手里有验证过的回滚之后,才碰生产。

可选辅助文件:`aids/worker-brief.md`、`aids/code-review-prompt.md`、`aids/worktree-lifecycle.md`、`aids/testing-anti-patterns.md`、`aids/debugging-techniques.md`。
