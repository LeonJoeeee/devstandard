# DevStandard(core.md 中文对照)

> 本文件供人阅读。插件注入 session 的是英文原版 [`core.md`](core.md),两者内容一致,以英文版为准。

这里的每个项目如何开发:完整生命周期何时启用、任务怎么执行、并行工作怎么协同。模板和辅助工具都在本插件里——需要时才读,绝不预读。

## 触发

- 人要求**新建仓库** → 启动完整生命周期,所有项目一致,没有大小例外:
  PRD → 架构文档 + ADR 日志 → 薄骨架(接口/边界落成代码,钉死并行工作的接入位)→ CI + 发布流水线 → 拆任务、扇出并行 session。
  到哪个产物再读 `howto/prd.md`、`howto/architecture.md`、`howto/adr.md`、`howto/cicd.md`——别提前读。
- 已有仓库里的任何改动 → 就是一个任务(规则见下)。没有项目级仪式。

## 执行一个任务

**写任何代码之前:先定下机器可自判的完成标准**——什么能机械地证明这个任务做完了(测试通过 / bug 不再复现 / 指标动了)。需求含糊 → 先和人定清楚。

**挑能托住这个活的最便宜档位:**
1. session 里直接干——多数工作的默认。
2. 1–3 个干净上下文的子 agent——有独立的块、或独立评审有帮助时;无循环、无宽扇出(子 agent 可以继续分包——对一个块的深度支援仍算本档)。
3. 一发小 workflow——只为真实的扇出(评审团)或真实的循环(修到全绿)。
4. 多发链式 workflow——工作跨越了决策点。

**每个档位都守的纪律:**
- 非平凡的设计,实现之前至少被一个干净上下文的评审驳过。
- 同一时刻只有一个写者。并行花在评审/验证上,绝不花在并发编辑上。
- "做完了"必须带证据:命令、退出码、输出。评审者失联算失败,不算通过。
- 只在改动触及顶层设计、或行动花费巨大时问人;其余自治。拿不准就当大事,问。

**workflow(档位 3–4):**一发 run 是一段盲飞的完整阶段——运行中无人能干预。发射前把花费封顶:评审团人数写死、每个循环带轮数上限、预算守卫。run 在决策点/检查点处拆,绝不因容量拆;run 之间靠磁盘上的 commit 和文档接力。模型按角色路由:判断/综合 → 可用的最强档;评审团 → 降一档;机械活 → 降两档。绝不从顶配 session 发射宽扇出——agent 会继承 session 的模型。

## 协作标准

- 一个任务 = 一条分支 = 一个 worktree = 一个 session。开工前:读仓库的 `docs/architecture.md`(共同底图),扫一眼 `docs/adr/`。
- 交付 = 推分支(项目用 PR 就开 PR)。**merge 归 main**:CI 必须绿。绝不从任务 session merge 或直推 main。
- 要动核心架构?绝不悄悄动:摆到台面 → 公开 merge → maintainer 批准 → 更新 `docs/architecture.md` + 写一篇 ADR。你认为底图本身错了,也走同一条路挑战——绝不闷头照着错的做。
- 安全底线:改动线上服务必须走分支 + 测试门 + 人工评审;绝不在生产数据上跑迁移——只在副本上。
- 本插件的可选辅助:`aids/code-review-prompt.md`、`aids/testing-anti-patterns.md`、`aids/debugging-techniques.md`。
